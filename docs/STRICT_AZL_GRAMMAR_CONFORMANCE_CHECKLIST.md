# Strict AZL Grammar Conformance Checklist

This checklist defines release-blocking grammar and runtime conformance rules for canonical native AZL execution.

## Scope

- Canonical runtime bundle generated by `scripts/start_azl_native_mode.sh`.
- Active native release gates under `scripts/run_tests.sh` and `scripts/run_all_tests.sh`.
- Production `.azl` components used in the canonical startup path.

## Release-Blocking Rules

1. Native runtime must boot and expose healthy API endpoints.
2. Combined runtime bundle must not contain host-language class syntax (for example `class Name {`).
3. Combined runtime bundle must not contain JS host syntax (for example `=>`, `constructor(`, `Math.random(`, `import ... from ...`).
4. Combined runtime bundle must not contain Python host syntax (for example `def`, `lambda`, `__name__ == "__main__"`, `print(`).
5. Legacy entrypoints must remain blocked under `AZL_NATIVE_ONLY=1`.
6. LHA3/quantum stack components must be present in canonical combined runtime.

## Validation Commands

- Full strict suite:
  - `bash scripts/run_all_tests.sh`
- Native checks only:
  - `bash scripts/run_tests.sh`
- Direct grammar gate:
  - `bash scripts/verify_azl_grammar_conformance.sh`

## Operational Notes

- Grammar conformance is validated against the live combined runtime path, not just source files in isolation.
- Any host-language syntax hit in the combined runtime is a release blocker and must be removed or excluded from canonical startup.
- If verification fails, fix source inclusion/routing first, then rerun:
  - `bash scripts/check_azl_native_gates.sh`
  - `bash scripts/verify_native_runtime_live.sh`
  - `bash scripts/verify_azl_grammar_conformance.sh`
